# Lawnmower Game — Claude Context

## Project
Pixel art top-down lawnmower game. Phaser 4.2.1, vanilla JS, no build step. Deployed to GitHub Pages via GitHub Actions on every push to main.

## Design Philosophy
This game is meant to be **relaxing and mindless** — a low-stakes activity, not a challenge. It should never cause frustration or invite competitiveness. Every design decision, including new feature requests, should be weighed against this before anything else:
- **No pressure.** No timers, scores, leaderboards, difficulty settings, or fail states. Nothing to lose, rush, or optimize. `WIN_PCT` exists so a level has an endpoint, not so the player is being judged.
- **No punishing mechanics.** Obstacles block movement but never undo progress — nothing reverts mowed grass, costs the player anything, or requires precision under pressure. (Sprinklers were removed specifically because they didn't fit this — see "No Sprinklers" below.)
- **Ambient, not adversarial.** Wildlife (squirrel, bird, deer, fox, dog, frog) exist for charm and gentle surprise, not to threaten or challenge the player — none of them can end a level badly or need to be "beaten." The dog scampers, it doesn't chase; the frog hides, it doesn't harass.
- **Gentle audio, gentle pace.** See the Audio section's "mindless peacefulness" tone — sounds are soft accents, not alerts or stingers. Movement speed is fixed at a comfortable medium (see "No Toggle-able Settings"), not something to min-max.
- **When in doubt, simplify toward calm.** If a proposed feature could read as stressful, punishing, or competitive (even subtly), that's a reason to reconsider or soften it before building it, not a detail to fix later.

## Key Files
- `game.js` — all game logic (single file)
- `index.html` — canvas host + DOM UI overlay + loading screen
- `levels/level-0N.json` — level maps (T=tree, G=garden, B=bush/hedge, P=pond, R=boulder, C=cactus, N=corn, .=grass); optional fields: `mowPattern`/`mowContrast` (see Mow Patterns), `theme` (see Border Themes), `fillBands` (see Grid Constants — portrait fill). **File name ≠ play order** — `levels/index.json` lists them in actual play order (currently staggered so no two forest-family levels are ever adjacent: `level-01, level-05, level-02, level-06, level-03, level-07, level-04`), and each file's own `name` field ("Level N") is kept in sync with its *position* in that list, not its filename — `BootScene` loads files in `index.json`'s order and assigns `level: arrayIndex`, and the in-game `L1`/`L2`/… HUD label (`hud-level`) is derived from that same array position (`this.currentLevel + 1`), not from `name`. Reordering `index.json` again later means re-syncing every reordered file's `name` field to match its new position.
- `vendor/phaser.min.js` — Phaser 4.2.1, vendored locally (not CDN) to avoid a third-party DNS/TLS roundtrip on load. To bump the Phaser version, manually re-download and overwrite this file.

## Loading Screen
`index.html` shows a pure CSS/HTML spinner (`#loading-screen`) on first paint, with no JS dependency — it's already in the markup before any script runs. `GameScene.create()` in `game.js` hides it (`.hidden` class) as its last step, once the scene is actually playable. If you add new async setup to `create()`, keep the hide call last.

## Grid Constants
```
CELL = 16px
YARD_X = 3, YARD_Y = 5 (border offset)
YARD_COLS = 18 (fixed — matches every level's authored width)
BASE_YARD_ROWS = 8 (authored level height)
```
`YARD_ROWS` (and so `ROWS`/`H`) is **computed at load** by `computeYardRows()` — on portrait + touch devices it grows past 8 to fill available vertical space (screen height minus `CONTROL_RESERVE_PX`, ~1 inch, reserved for the D-pad), clamped to `MAX_YARD_ROWS`. Desktop and mobile landscape always get the fixed 8 rows. This is a one-time computation at page load; it does not live-recompute on resize or orientation change. `normalizeMap()` pads each authored level's map out to `YARD_ROWS` — level files themselves stay a fixed 18×8 and never need to change. Originally the padding was all plain grass with the authored map centered, which on tall portrait grids (a typical phone computes ~30 rows) left every obstacle clumped mid-screen with huge bare bands above and below — reported directly as "everything is kind of clumped in the middle." Now the padding is filled with the level's `fillBands` (optional per-level JSON field: an array of 2-row, 18-col band patterns, theme-matched), stamped outward from the authored map's top and bottom edges in a `[2-row gap][2-row band]` rhythm cycling through the list (bottom side starts one band later so the sides don't mirror), sub-4-row remainders staying plain grass at the outer edges. The rhythm preserves the ≥2-cell separation convention automatically (band↔band exactly 2, band↔authored ≥3 via the authored maps' own open edge rows); fill bands must never contain `P` (multiple ponds would break the single-pond `pondBounds` assumption). Landscape/desktop (`YARD_ROWS` = 8) is a no-op, so those layouts are untouched. A Python replica of the fill logic validated every level at every height 8-60: rectangular clusters, even T/G dims, no squeezes, single pond. Only rows grow (never columns), so the mower stays a fixed single-cell size — there's no swath-widening or world-scaling here, by design (see git history around 2026-07-18 for why that combination was reverted).

`YARD_Y` (5) and `BASE_YARD_ROWS` (8) were chosen together so `ROWS` on desktop/landscape (`= BASE_YARD_ROWS + YARD_Y*2` = 18) — and so the overall canvas height `H` — stays exactly what it was before the yard was shortened (`YARD_Y` 3, `BASE_YARD_ROWS` 12, same `ROWS` of 18). Shortening the mowable area is purely a matter of shifting where the yard/border boundary falls within an unchanged canvas, not shrinking the game window. All three level files were re-authored to exactly 8 rows (obstacle clusters redesigned to keep ≥2-cell gaps between them) rather than relying on `normalizeMap()`'s generic crop-to-fit, so every level's layout is exactly what's on screen.

## Phaser 4 Gotchas
- **RenderTexture**: must call `.render()` after all `.stamp()` calls or nothing appears
- **Full-screen RT**: always `rt.setOrigin(0, 0)` — default origin is center which offsets everything
- **Off-screen graphics**: use `this.make.graphics({ add: false })` then `.generateTexture()` then `.destroy()`
- **`pixelArt: true`** in game config — critical for crisp sprites, also sets `roundPixels`
- **Scale / DOM overlay**: use `this.scale.canvasBounds` (not `canvas.getBoundingClientRect()`) to position the DOM overlay; hook `this.scale.on('resize', ...)` to reposition on window resize
- **Depth**: all GameObjects default to depth 0 and render in creation order; use `setDepth()` to reorder
- **Multi-touch**: set `input: { activePointers: 3 }` in game config
- **Scene restart persistence**: module-level `let g_foo` survives `scene.restart()`; `this.foo` does not

## DOM Overlay Pattern
UI text lives in `#ui-canvas` (a `position:absolute` div over the canvas) to avoid the `image-rendering: pixelated` CSS blurring canvas text. `syncUIOverlay()` reads `canvasBounds` and applies a CSS `scale()` transform to match the Phaser FIT scale.

## Decorative Border (and Border Themes)
`buildBackground()` branches on the level's opt-in `theme` field: the default (no field) is the forest border described below; `"theme": "mountain"` (`level-05.json`) swaps the tree/pine rows for an outer rim of big snow-capped peaks (`bg_mountain`) with smaller snowless foothills (`bg_mountain_small`) staggered between/inside them and a sparse `bg_pine` treeline on the inner top/bottom rows — same outer/inner layering and shared y-sort as the forest theme, same placement guards. The wild-grass base, accent blades, and wildflowers are shared by forest/mountain (they read as alpine meadow just as well). Mountain levels also get tiny alpine flowers (white/purple/yellow 2px dots) scattered through the yard's own unmowed grass, baked into the depth-0 background like the blades so mowing over a cell covers them. The mountain theme also forces every in-yard tree cluster to `tree_evergreen` in `buildObstacleLayer()` (no random round/deciduous picks — alpine treeline), and swaps which border peeker spawns (goat instead of deer+fox, see Goat below).

`"theme": "desert"` (`level-06.json`) goes further: the border base becomes muted sand (`0x6e5a32` — still darker than the yard, same lawn-pops principle), the wild grass blades become sparser/shorter sun-bleached scrub tufts, and the wildflower palette shrinks to sun-washed desert blooms (coral/yellow/white) at half the rate — sparse is the desert look. Border rows: flat-top buttes (`bg_butte`, the big far landform), tall saguaros (`bg_saguaro`) staggered between/inside, prickly pear clusters (`bg_prickly`) with occasional saguaros nearest the yard — same layering/y-sort/guards as the other themes. The yard itself stays green on every theme: an irrigated lawn holding out against the landscape. Desert wildlife: fox only as peeker (kit fox reads fine in scrubland, a deer doesn't) plus the rolling tumbleweed (see Tumbleweed below); birds/squirrel/dog unchanged.

`"theme": "farm"` (`level-07.json`) swaps the border for a working farmyard: the base becomes dirt/hay brown (`0x5a4a26`), and wild grass becomes dry golden hay tufts — a farm border reads as an unmown hayfield, not sparse like desert. Unlike every other theme, `bloomChance` is forced to `0`: no wildflowers at all, since the border's already busy with fence/barn/haybale/corn decor and flowers dotted between them read as clutter rather than an accent (the yard's own garden bed still supplies color) — reported directly as wanting the flowers taken out on the sides. Unlike the other themes, farm doesn't follow the outer-landmark/mid-scatter/inner-treeline layering: a wooden fence (`bg_fence` horizontal on top/bottom, `bg_fence_v` vertical on left/right) rings the **entire** yard right at its edge on all 4 sides — a real field is fenced all the way around, not just top/bottom — and exactly **2** barns (`bg_barn`, fixed positions, not a spaced loop — a real farm has one or two, not several; an earlier pass placed them at mountain-peak/desert-butte density and it read as "too many barns") are the only landmarks. Everything else in the border is a random mix of hay bales (`bg_haybale`) and border-scale corn stalks (`bg_corn`, bigger than the in-yard 16×16 `corn` texture, same relationship as `bg_saguaro` vs. the in-yard cactus) filling the outer rim. `bg_corn` is 3 stalks of staggered height with 4 splayed leaves and 2 cobs (one partly tucked behind the tallest stalk, darker so it reads as sitting further back) — a first pass with 2 thin stalks and a single cob read as sparse next to the bulkier hay bales and fence rails it's mixed among, reported directly as wanting the corn along the sides improved. Farm wildlife: the cow only as peeker, replacing deer+fox (see Cow below); birds/squirrel/dog unchanged.

The unmowable border margin (outside `YARD_X`/`YARD_Y`) is dark wild grass (`0x1e3a12`) with denser/taller accent blades than the yard's own, small scattered wildflowers (same sampling loop as the blades, ~10% chance per sample point, random pick from a small color palette), plus a bigger, more muted `bg_tree` texture scattered around all 4 sides — purely cosmetic, baked into the depth-0 background RT in `buildBackground()`, no collision (the border was already unreachable). `bg_tree` (and `bg_pine`) are generated inline in `buildBackground()` itself rather than in `buildLevelTextures()`, since the latter runs later (from `buildObstacleLayer()`). `bg_pine` (conical, same muted palette) is staggered in alongside `bg_tree` on all 4 sides — offset half a step along the row and tucked closer to the outer edge — for a layered forest-edge look rather than a single flat row of one shape. All tree placements (`bg_tree` and `bg_pine`, every side) are collected into one array and y-sorted before stamping (painter's algorithm) rather than stamped in a fixed type order — with a fixed order, whichever type stamps last always draws on top of the other wherever they touch (e.g. a pine's trunk poking out over a round tree's canopy), regardless of how tight the jitter/spacing is tuned to reduce overlap in the first place.

The top/bottom sides get a **second, inner** tree/pine row (`innerTreeY`/`innerTreeYB`/`innerPineY`/`innerPineYB`, offset in from `yardT`/`yardB` rather than the canvas edge) alongside the original outer row (fixed 20px from the canvas edge, `margin`). The outer row alone was deep enough to reach the yard boundary back when the border was shorter (`YARD_Y` 3) — a single ~44px-tall tree centered 20px from the edge already spans almost the full old 48px margin. Taller borders (`YARD_Y` 5, 80px margin) without a second row would otherwise leave a bare gap of just grass/wildflowers between that row and the yard; the inner row (skipped entirely if the border's too short to fit one without the two colliding, via an `innerTreeY > margin + 30` guard) fills that gap.

## Obstacle System
- `obstacleGrid[r][c]` — gardens, bushes/hedges, and ponds (full block); trees don't set this
- `trunkPositions[]` — tree trunk pixel-radius collision (6px); player can enter and mow the cell but can't pass through
- `obstacleClusters[]` — gardens **and** bushes/hedges only (not ponds); auto-mow cells when all perimeter cells are mowed (`this.totalCells` is `YARD_ROWS * YARD_COLS` minus `this.pondCellCount` — no bush-cell subtraction needed there, since bushes reach mowedCount the same way gardens do, but pond cells never do, so they're excluded from the denominator instead)
- `isNearGarden()` — despite the name, this is the general obstacle-edge collision check (applies to any `obstacleGrid`-blocked cell — gardens, bushes, and ponds alike): it samples a small cross of points around the player (±6px, matching the mower's visual half-width) rather than just the exact center, so the mower's sprite stops right at the edge instead of visually overlapping into it before the single tracked point crosses the cell boundary
- **Corner-escape nudge**: `update()`'s movement resolution normally tries full diagonal → horizontal-only → vertical-only, in that order, and stops there. Approaching a concave obstacle corner (a garden/bush/pond's outer corner) diagonally can land the player within `isNearGarden()`'s ±6px margin of *both* the horizontal and vertical boundary at once — a spot where the horizontal-only and vertical-only slides are *also* both blocked, since each one's cross-sample still lands in-margin on the other axis. That's a genuine mutual deadlock, not just an overly strict margin: confirmed empirically (frame-by-frame position tracing) that shrinking the margin fixes it for some corners/approach-angles but leaves others (e.g. a garden's bottom corners under one approach vector stayed fully stuck even at margin 3) — so margin tuning alone isn't a reliable fix. Instead, when all three slides fail and the player is still holding a direction, a 4th fallback nudges the player a fixed 7px (margin + 1, so it clears the dead zone in one step) toward whichever of the 4 cardinal directions is actually open, checked in a fixed up/down/left/right order. 7px was chosen over a smaller nudge (2px was tried first) specifically to avoid jitter: a nudge smaller than the margin doesn't fully clear it, so continued held input immediately re-blocks on the very next frame, reading as a visible vibration right at the corner — worse than the freeze it was meant to fix. Verified via Playwright: all 4 corners of a test garden, plus a bush corner and non-45° approach angles, now slide through to the yard boundary with exactly one clean nudge and no oscillation.
- `TREE_TYPES` (`tree_round`, `tree_evergreen`) — one is picked at random per contiguous tree cluster in `buildObstacleLayer()` (not per 2×2 sub-block), so a single clump of trees reads as one coherent species. Both keep the trunk in the same local y=18..31 footprint so the shared `ty + 8` trunk-collision offset works for either without a per-type adjustment. (A third variant, `tree_willow`, was tried and dropped — didn't read well; see git history around 2026-07-18 if revisiting.)
- Bushes/hedges (`B` in level maps) stamp the single `bush` texture **per cell** (16×16, not the 32×32 2-cell blocks trees/gardens use), since a hedge is typically a single-cell-wide row of arbitrary length rather than a 2×2-aligned cluster. They share the exact same cluster/perimeter/auto-mow code path as gardens in `buildObstacleLayer()` (`type === 'G' || type === 'B'`) — the grass underneath turns mowed once the player's fully mowed around them, just hidden under the bush texture (obstacle layer depth 3, above mowed layer depth 1) same as gardens
- Ponds (`P` in level maps, introduced with `level-04.json`) also stamp per-cell like bushes, but are **permanent water** — no cluster/perimeter tracking, so they never auto-mow no matter what's mowed around them. `buildObstacleLayer()` tracks `this.pondCellCount` (excluded from `totalCells`) and `this.pondBounds` (the cluster's bounding box, assumes a single contiguous pond per level — used to place the frog, see below). A few random cells get a `lilypad` texture stamped on top (~18% chance) for visual variety instead of a flat uniform tile.
- Boulders (`R` in level maps, introduced with `level-05.json`'s mountain valley) stamp the per-cell `boulder` texture and get the same **permanent** treatment as pond water — full `obstacleGrid` block, no cluster/perimeter tracking, never auto-mow, cells excluded from `totalCells` via `this.rockCellCount` — just dry: no bounds tracking or resident critter. Rock formations can be any rectangle (2×1, 1×2, 2×2, single) since per-cell stamping doesn't need 2×2 alignment; the corner-escape nudge works on their corners the same as any other `obstacleGrid` obstacle (verified with diagonal approaches against both a 2×2 pile and a 1×1 rock).
- Cacti (`C` in level maps, introduced with `level-06.json`'s desert) stamp the per-cell `cactus` texture but share the garden/bush **cluster/auto-mow** path (`type === 'G' || 'B' || 'C'`), not the pond/boulder permanent path — the grass under a cactus turns mowed once the player's mowed all the way around it, hidden under the cactus texture, so cactus cells count toward `totalCells` like bush cells do. Brighter green than the border's muted `bg_saguaro`, same in-yard-vs-scenery brightness logic as bush vs `bg_tree`.
- Corn (`N` in level maps, introduced with `level-07.json`'s farm) stamps the per-cell `corn` texture and shares the same garden/bush/cactus **cluster/auto-mow** path — mowing all the way around a corn patch "harvests" it, same mechanic as clearing a cactus or hedge, just themed as a crop instead of scenery.

## Mowed Grass Rendering
Mowed grass is **blocky and grid-aligned, one full cell at a time**. `mowAt(px, py)` figures out which grid cell the player is in and, if it's not already mowed to this depth, stamps the single `mowed_H_full` (16×16, per deck height) texture at that cell's exact center via `this.mowedRT.stamp()`. There's only one texture shape — always full-cell, never a narrower mower-width variant — which is what makes this gap-free on any path (straight, diagonal, turn, parallel lanes) and impossible to bleed past the yard border: every stamp lands at exact grid coordinates already bounds-checked against `YARD_COLS`/`YARD_ROWS`.

This is a deliberate return to a simpler design after two failed attempts at matching the mower's exact sub-cell pixel width: (1) direction-dependent narrower textures (`_v`/`_h`) gapped on turns and parallel lanes since they didn't account for the player's actual sub-cell offset, and (2) a continuous stroke tracing the player's exact path frame-by-frame fixed those gaps but read as "too fluid"/not grid-aligned, and its stroke width could still creep a few px past the yard edge into the border independent of collision. Full-cell stamping sidesteps both: it's inherently blocky (matches the grid the whole game is built on) and inherently bounded (a full cell literally cannot render outside the yard).

`buildMowedTextures()` generates just this one `mowed_H_full` texture per deck height — used by both the player's own mowing and by `checkClusterCompletion()`'s garden auto-mow (which stamps whole cells instantly and is hidden under the garden's own obstacle-layer texture anyway, depth 3 above the mowed layer's depth 1).

### Mow Patterns (per-level)
`buildMowedTextures()` also generates a `mowed_H_full_alt` variant per deck height — the same texture with `shadeColor()` darkening its base/stripe colors by a level-tunable factor. `mowedTextureKey(h, gc, gr)` picks between the two based on the level's opt-in `mowPattern` string field (in that level's JSON — omitted/unrecognized means a flat, uniform mow, unchanged from before mow patterns existed). `mowAt()` and `checkClusterCompletion()` both call it instead of hardcoding `mowed_${h}_full`, so gardens/bushes auto-mow with the same alternation as the player's own mowing. Patterns so far:
- `'checker'` (`level-01.json`): alternates by `(gc + gr)` parity — a single-cell checkerboard.
- `'stripe'` (`level-02.json`): alternates by grid-column parity — vertical bands, mimicking the light/dark banding real mowers leave cutting adjacent passes in opposite directions.
- `'rings'` (`level-03.json`): alternates by Chebyshev ("square") distance from the yard's center cell (`Math.floor(Math.max(|gc - centerC|, |gr - centerR|))`), i.e. concentric square rings growing outward from the middle, alternating shade ring by ring — a target/bullseye look starting small and growing outward as more of the yard gets mowed.
- `'diagonal'` (`level-04.json`): alternates by `Math.floor((gc + gr) / 2)` parity — 2-cell-wide diagonal bands at 45°, same band thickness as `'stripe'` just along the other axis.
- `'hstripe'` (`level-05.json`): alternates by grid-row parity — horizontal bands, `'stripe'` turned 90°: passes running across the valley floor.
- `'bigchecker'` (`level-06.json`): alternates by `(⌊gc/2⌋ + ⌊gr/2⌋)` parity — a checkerboard of 2×2 blocks, coarser and calmer than `'checker'`.
- `'lattice'` (`level-07.json`): alternates whenever `gc % 4 === 0 || gr % 4 === 0` — a thin crossing grid every 4 cells, reading as a quilted/fenced-field lattice rather than solid bands or blocks.

`mowContrast` (a `levelData` field alongside `mowPattern`, e.g. `"mowContrast": 0.9`) is the `shadeColor()` factor for that alt texture — lower is darker/more pronounced, `1` would be no visible difference at all — read once in `buildMowedTextures()` and defaulting to `0.9` if the field is omitted. It went through a few rounds of tuning: 15% darker (0.85) was the original value, sized for `'stripe'`/`'rings'`'s coarse, wide bands; once `'checker'`/`'diagonal'` were added (alternating every 1-2 cells in both grid directions, not just one) that same contrast read as busy/noisy, so it came down to ~6% (0.94) — then that read as barely-there once compared side by side, so it settled at 10% (0.9) as a per-level field instead of a hardcoded constant, so any future pattern/level combination can be tuned on its own without a repeat of that back-and-forth. All seven levels currently use the same `0.9`.

## No Toggle-able Settings
Deck height, speed, blade, and distractions (squirrel) all used to be player-adjustable via a lever/toggle panel docked to the right border. That panel was removed — there is no in-game way to change these anymore. They're fixed at their old defaults: `this.deckHeight = 2` (set once in `create()`), `SPEED_STEP = 2` (medium), blade always on, distractions always on. If a "make it configurable again" request comes in, the lever UI code is recoverable from git history (commit `1b1fe23` and earlier had the full lever/toggle implementation).

**Easy Mode is a deliberate exception**, not a reversal of this — it's an accessibility toggle, not a difficulty/tuning knob like the removed levers. See below.

## Easy Mode
`#easy-toggle` (top-right of the canvas overlay, mirroring `#level-nav`'s top-left placement) lets the mower roll straight over every obstacle instead of getting stopped by it — added because a younger player kept getting physically stuck maneuvering around obstacle edges. Clicking it calls `setEasyMode()`, which flips module-level `g_easyMode` (global, not per-level — this is about the player, not the level) and persists it to `localStorage` (`mower_easyMode`, wrapped in try/catch since it can throw in some private-browsing contexts) so a parent doesn't have to re-enable it for their kid every session; `buildHUD()` re-syncs the button's `.active` visual state from `g_easyMode` on every level load since the setting itself isn't per-level state.

`isObstacle()` is the only thing it touches: the static-obstacle checks (`isNearGarden()` — garden/bush/pond/boulder/cactus edges — and the tree-trunk pixel-radius loop) are skipped entirely when `g_easyMode` is on, while the squirrel/dog cell-blocking checks below them stay active regardless — those are temporary (the animal moves on its own) and not a source of the same stuck-in-place frustration a static obstacle is. Nothing else changes: `mowAt()`'s own `obstacleGrid` check is untouched, so gardens/bushes/cacti still only auto-mow once their perimeter is actually cleared, ponds/boulders are still never mowed, and `totalCells`/`WIN_PCT` math is completely unaffected — verified via the same mow-to-100%-and-win regression with Easy Mode on, identical totals to normal mode on all 6 levels. It only removes the "getting stuck" friction, not any of the actual mowing mechanic.

## No Sprinklers (removed)
Sprinklers (a periodic pop-up spray animation that reverted a small mowed patch back to grass) were removed — buggy and the feel wasn't landing. The full implementation (`scheduleSprinkler`/`popSprinkler`/`findSprinklerPos`/`animateSprinkler`/`eraseMowedBlock`, plus the `activeSprinkler` collision/mow-blocking checks in `isObstacle()`/`mowAt()`) is recoverable from git history if revisiting (see commits around 2026-07-19). Squirrels are unaffected and still active.

## Mobile Layout
`#game-container` hosts the Phaser canvas (`scale.parent` in the game config); `#controls-spacer` (a sibling, sized via CSS flex) reserves real room for the D-pad on every touch device — below the canvas in portrait, beside it in landscape (`body`'s `flex-direction` flips from `column` to `row` via a `(orientation: landscape)` media query) — never a floating overlay on top of the canvas. `applyResponsiveLayout()` in `game.js` sets `#game-container`'s `aspect-ratio` from the computed `W`/`H` in portrait only, so Phaser's FIT scaling fills it with no dead space; landscape doesn't need this since its aspect ratio doesn't grow to fill the screen the way portrait's row-count does, so a plain fixed-width reserve (`190px`) on `#controls-spacer` is enough. The D-pad itself is one shared size (`174px`) for both orientations now, sized to fit that shared ~190px reserve — the D-pad's base CSS rule (fixed position, `216px`) is unreachable in practice on any real touch device, kept only as a fallback for a touch device matching neither media query.

This replaced an earlier design where landscape left the D-pad `position: fixed` over the canvas's assumed side "dead zone" (the empty margin FIT scaling leaves when the container is much wider than the game's 4:3 aspect). That assumption didn't hold on real phones or tablets: the dead zone's width depends entirely on the device's own aspect ratio, and on every phone tested (iPhone SE/14, Pixel 5) plus ~4:3 tablets (iPad mini, zero dead zone at all) the fixed 216px D-pad sat 70–234px into the mowable yard instead of beside it — reported directly as "controls overlaying the mowing area, mowing area off screen." A real reserved strip removes the guesswork: `#game-container` and `#controls-spacer` split the viewport by CSS flex instead of the D-pad guessing where empty canvas margin might be, so overlap is structurally impossible regardless of device aspect ratio. Verified across four device sizes (Playwright, touch+landscape emulation): zero canvas/D-pad overlap, canvas fully onscreen, and a real D-pad tap still moves the player through the actual game loop.

## HUD
No progress bar — just the level indicator (`#hud-level`, still in the canvas-scaled `#ui-canvas` overlay, with a `text-shadow` for contrast now that there's no background bar behind it) and a plain percentage readout (`#hud-pct-bottom`) docked near the D-pad, outside the canvas entirely, as a sibling of `#dpad` inside `#controls-spacer`. Default (desktop mouse/trackpad, D-pad hidden entirely): `position: fixed` bottom-right, the same technique `#dpad` uses to float free of `#controls-spacer`'s zero-height box in that mode. Portrait: `position: absolute` pinned to the right edge of `#controls-spacer` (still `position: relative` in that mode) rather than sitting in the flex row next to the D-pad — keeps a clear gap between them instead of a small margin, and lets the D-pad center on its own. Landscape: the same idea rotated 90° — pinned to the strip's top edge instead of its right edge, with `#level-reset` mirroring it at the bottom edge, D-pad centered between the two.

`#hud-level` sits inside a `#level-nav` flex row alongside `#level-prev`/`#level-next` arrow buttons, so they stay adjacent regardless of the level text's width (`L1` vs `L10`). `#ui-layer` (the whole canvas-scaled overlay) is `pointer-events: none` so clicks fall through to the canvas underneath everywhere else, but `#level-nav` opts back into `pointer-events: auto` so its buttons are actually clickable. `buildHUD()` wires `.onclick` on both (reassigned, not `addEventListener`, since these DOM elements persist across `scene.restart()` — same reasoning as `winEl.onpointerdown`) and sets `.disabled` from `this.currentLevel` bounds every time it runs (i.e. every level load): `level-prev` disabled at the first level, `level-next` disabled at the last, no wraparound. `goToLevel(index)` — used by both buttons — re-checks those same bounds itself (`index < 0 || index >= this.allLevels.length`) before calling `scene.restart()`, so it's safe even if called from somewhere that doesn't already respect the buttons' disabled state.

`.level-nav-btn`'s base size (~22×20px, bumped up a notch from an original ~18×16px that read as too small/fiddly to reliably click) is fine for a mouse but too small a touch target on its own — a `(hover: none) and (pointer: coarse)` media query bumps both buttons up specifically for touch, since the whole `#ui-canvas` overlay these buttons live in is already CSS-scaled to match the canvas (see DOM Overlay Pattern), so a fixed logical size renders at a fairly consistent physical size across devices regardless of viewport. Sized at 28×28px (also bumped up from 24×24px alongside the desktop size) — an earlier pass went to the full 44px touch-target guideline (42×42px) but read as too visually heavy next to the small level text, so this trades strict guideline compliance for a size that stays out of the way. `#level-reset` (46×46px, see below) wasn't part of that complaint and is unchanged.

`#level-reset` (the `↺` button, a sibling of `#dpad`/`#hud-pct-bottom` inside `#controls-spacer`, using the exact same float-free-of-the-zero-height-box technique) discards the current level's saved progress and restarts it — `resetLevel()` just deletes that level's `g_levelState` entry (see below) before `scene.restart()`, so the level comes back in its normal fresh-start state rather than needing separate reset logic.

### Preserving Level State Across Navigation
Since `#level-prev`/`#level-next` (and normal win-triggered `advanceLevel()`) all go through `scene.restart()`, mowing progress would otherwise be lost every time — module-level `g_levelState` (an object keyed by level index, survives `scene.restart()` the same way `g_introShown` does) is a snapshot of everything needed to resume: the mowed-depth grid, `mowedCount`, `won`, each obstacle cluster's `done` flag, player position/facing, and the dog/frog's state. `goToLevel()` and `advanceLevel()` both call `saveLevelState()` right before `scene.restart()`.

`create()` checks `g_levelState[this.currentLevel]` after `setupPlayer()`/`buildObstacleLayer()` run (so there's a grid and obstacle clusters to restore *into*) and, if present, copies the saved values back in — including re-stamping every already-mowed cell onto the (freshly blank) mowed-grass RT, since the grid array only tracks *state*, not the pixels that make it visible. A restored `won` deliberately does **not** call `showWin()` — the win overlay is a full-screen `pointer-events: auto` div sitting above `#level-nav`, so popping it back up on a mere revisit (rather than an actual live win) left no way to click `#level-prev`/`#level-next` without tapping through the overlay first, which silently called `advanceLevel()` and skipped a level — the exact "finished level 2, tried to skip ahead, and it acted like Level 4 was already done" bug this fixes. `this.won` is still restored `true` (freezing movement and showing 100%/the fully-mowed yard), just without the modal. `resetLevel()` is the escape hatch — `delete g_levelState[this.currentLevel]` before restarting means that level's next `create()` finds nothing to restore and starts clean, same as a level visited for the first time ever.

`showWin()`'s "All levels complete!" message is decided by `allLevelsComplete()` — every level index other than `this.currentLevel` (which is implicitly done; `showWin()` only ever fires with `this.won` already true) must have `g_levelState[i].won` — rather than the position-only check it used to be (`currentLevel >= allLevels.length - 1`). That older check meant jumping straight to the last level via `#level-next`/`#level-prev` and winning it claimed every level was done even with the rest untouched — reported directly as "skip to the last level, finish it, and it says all levels done." The non-complete branch names whichever level `advanceLevel()`'s own `(currentLevel + 1) % length` wraparound will actually go to next (reusing that exact formula) rather than assuming the sequential `currentLevel + 1`, so it correctly reads "Up next: Level 1" instead of a false "All levels complete!" in that same skip-ahead scenario. Verified via Playwright across three orders: skipping straight to the last level first (no false completion claim), normal sequential 1→6 play (unchanged messages throughout), and finishing levels out of order with the last one mowed being a non-final level positionally (correctly still reports "All levels complete!").

## Intro Overlay
`#intro-overlay`/`#intro-box` mirror the win-overlay pattern (same dark rounded box style) but with an explicit `#intro-start-btn` button instead of tap-anywhere, and is visible by default in the HTML markup (not toggled by JS) so it's already up the instant the loading screen hides. Shown once — `buildIntroOverlay()` early-returns if the module-level `g_introShown` flag (persists across `scene.restart()`) is already true, and the Start button / Enter / Space handler sets it. `this.started` (gated in `update()`'s early-return alongside `this.won`) blocks all movement/mowing until Start is tapped; `create()` seeds it from `g_introShown` so every level after the first starts immediately with no modal. `scheduleSquirrel()`/`scheduleBird()`/`scheduleDeer()` are deferred to the Start handler on the very first level (rather than called unconditionally in `create()`) so their timers don't start counting down while the modal is still up.

## Birds, Deer, Fox, Goat, Cow, and Tumbleweed
Purely cosmetic, no collision, no per-level cap (unlike squirrels) — same `schedule*`/`launch*`/`update*`/`draw*` structure as the squirrel, but all are confined to the border margin and never enter the yard. Birds (`scheduleBird`/`launchBird`/`updateBird`/`drawBird`) pick one of the 4 border strips and fly straight across it (a 2-line wing-flap silhouette, light-colored — a dark one barely shows up against the similarly-dark wild grass border — alternating every ~120ms, plus a small sine-wave flutter perpendicular to travel that's purely visual and doesn't affect the tracked x/y) every 15-30s, despawning once it's well past the far edge, on every level regardless of theme. Deer and fox (`scheduleDeer`/`scheduleFox` etc.) peek out from the left or right border independently of each other (either, both, or neither can be active at once) on their own random interval: a 3-phase `peek`/`hold`/`retreat` state machine (`this.deer.t`/`this.fox.t` goes 0→1→0) slides a small body partway in from off-canvas, holds, then retreats — the peek depth is capped well short of the yard boundary. Deer use the same muted palette as `bg_tree`/`bg_pine`; the fox is rust-orange with a pale chest/tail-tip so the two read distinctly from each other at a glance.

The goat (`scheduleGoat`/`launchGoat`/`updateGoat`/`drawGoat`) is the mountain theme's peeker — same peek/hold/retreat machine, popping out from behind the peaks on the left/right, with a quicker peek and longer hold than the deer (nosy, sure-footed). Cream-white body with backswept dark-gray horn nubs and a muzzle/beard patch, so it reads distinctly against the gray peaks. `schedulePeekers()` (the single scheduling entry point called from both `create()` and the intro Start handler) decides which theme wildlife a level gets: mountain levels get the goat **instead of** the deer and fox (forest wildlife wandering out of a rock face would break the scene), desert levels get the fox only (kit fox) plus the tumbleweed, farm levels get the cow only (see below), all other levels get deer+fox.

The cow (`scheduleCow`/`launchCow`/`updateCow`/`drawCow`) is the farm theme's peeker — same peek/hold/retreat machine, popping out from behind the barn/fence. The slowest peek and longest hold of any peeker (1100/2800/1100ms vs. the goat's 700/2600/700ms): cows are placid, unhurried animals, which fits the game's gentle pace better than a quick pop. White Holstein body with irregular black patches and a pink muzzle — instantly reads as "cow" against the farm border's warm browns, distinct from every other peeker's palette.

The tumbleweed (`scheduleTumbleweed`/`launchTumbleweed`/`updateTumbleweed`/`drawTumbleweed`) is desert-only ambience — structurally a ground-level bird (same schedule/launch/despawn cycle, no collision, no effect on mowing) that rolls all the way across the yard every 9-20s: constant horizontal velocity, a skipping `|sin|` bounce off a fixed `baseY`, and a ground shadow that stays at `baseY` fading as it hops. Drawn as a scraggly ball — outer ring plus 3 spokes rotated by a phase accumulator, so it visibly tumbles rather than slides. Unlike every other critter it crosses the yard itself (not the border): rolling right over the lawn is exactly what tumbleweeds do, and with no collision or mow effect it stays harmless.

## Dog
Unlike bird/deer/fox, the dog is **in the yard itself** (not the border) and present for the whole level rather than scheduled — it's initialized once in `create()` via `pickDogSpot(this.player.x, this.player.y, 80)`, which scans `this.levelData.map` for plain-grass (`'.'`) cells at least the given pixel distance from a point and returns a random match's cell-center (or `null`, falling back to the yard's bottom-right corner). It has real collision, mirroring the squirrel's exact-cell block in both `isObstacle()` and `mowAt()` (blocks only the dog's current grid cell, always, no `.active` flag needed since the dog always exists).

`updateDog(dt)`, called every frame from `update()` (no `schedule*`/timer — it reacts to proximity, not an interval), is a 2-state machine:
- **idle**: once any post-flee cooldown (`cooldownRemaining`, 1500ms) has elapsed, checks `Math.hypot()` distance to the player against `TRIGGER_DIST` (48px, ~3 cells); if within range, calls `pickDogSpot(this.player.x, this.player.y, 100, this.dog.x, this.dog.y)` — deliberately keyed off the *player's* position rather than the dog's own old spot, so the destination is guaranteed far enough from the player to not immediately re-trigger — and switches to `fleeing`, plus fires `playDogBark()`.
- **fleeing**: linearly interpolates (`Phaser.Math.Linear`) from the old spot to the new one, plus a small perpendicular sine wobble (fading out near the end) so it scurries rather than slides in a dead-straight line. Duration is **speed-based** (`fleeDist / FLEE_SPEED`, 55px/s, clamped 400–1400ms) rather than a fixed time, so near and far scampers both move at the same hurried-but-not-blurring pace — noticeably under the player's own 80px/s medium speed, which is what reads as a "scamper" instead of a dash. (An earlier fixed-400ms version covered any distance in the same time, so longer flees blurred past at 250px/s+ — the reported "moving fast" issue.)

On a pond level, `pickDogSpot()`'s scamper destination also needs to keep the dog from cutting straight across the water — a straight-line-crossing test (mirroring the squirrel's) turned out too weak here: `level-04.json`'s pond spans nearly the entire yard along one axis, so there are stretches where *every* straight line between two valid grass cells crosses it, and a check that only rejects crossing paths eventually has nothing left to accept and falls back to one anyway. Instead, candidates are first restricted to the *same side* of the pond as the dog's current position (left/right of its column span, or above/below its row span) — two points both on one side of a dividing line can never have a straight line between them cross to the other side, which holds regardless of how much of the yard the pond eats up. The "far enough from the player" distance preference is applied only *within* that same-side set (falling back to the wider, distance-unfiltered same-side set if none qualify, and only falling back further to every side if the dog's own side somehow has no candidates at all) — so a dog boxed into a narrow strip degrades to a closer-than-ideal escape rather than ever crossing the water to satisfy the distance check instead.

`drawDog()` switches pose by state rather than using one silhouette throughout:
- **idle**: a sitting silhouette — low, wide haunches (`fillEllipse`) with the chest/head held upright above them and the tail curled in at the side, distinctly "at rest" rather than the running pose standing still.
- **fleeing**: the original low, stretched pose with dx/dy-driven head placement (same technique as the squirrel) and a 1px vertical bob, for a scampering read while actually moving.

Palette is warm brown (`FUR`/`FUR_DARK`) with a tan muzzle patch, not flat near-black — the original near-black fur plus straight-up-pointing ears read as a rabbit rather than a dog. Fixed with two changes together: floppy triangular ears (`fillTriangle`) drooping outward from the sides of the head instead of straight rects on top, and a lighter muzzle/snout patch on the head to break up the round-head silhouette.

`playDogBark()` (module-level, alongside the other audio functions below) is a two-note "arf-arf" — `square` oscillators (woofier than a raw sawtooth) each swept 400Hz→150Hz, run through a `BiquadFilterNode` lowpass that sweeps down alongside the pitch (2000Hz→500Hz) to round off the harsh top end, with a fast attack and quick exponential decay. (An earlier single-note raw-sawtooth version read as more of a buzz than a bark.)

## Frog and Pond
Introduced with `level-04.json`. Only exists on levels with a pond (`this.pondBounds` set — `this.frog` stays `null` otherwise, and `updateFrog()`/`drawFrog()` both early-return on that). Unlike the dog, it has **no collision** (purely cosmetic, like the bird/deer/fox) and doesn't relocate — it always jumps to and from the same two points.

`pickFrogSpot()` (called once from `create()`) finds where the frog sits: tries the plain-grass (`'.'`) cell immediately left of the pond's bounding box first, then right, then top, then bottom ("a frog on the side" — side tried before top/bottom), paired with the adjacent pond-edge cell it'll jump into. Returns `null` (no frog) if every side is blocked, rather than crashing.

`updateFrog(dt)`, called every frame from `update()` (no timer, reacts to proximity), is a 4-state machine:
- **sitting**: once any post-jump cooldown (`cooldownRemaining`, 900ms) has elapsed, checks distance to the player against `TRIGGER_DIST` (40px); if within range, switches to `jumping` and fires `playSplash()`.
- **jumping**: linearly interpolates from its resting spot to the pond-edge cell over `HOP_MS` (350ms), with a `Math.sin(π·t)` arc (`hop`) applied as a y-offset in `drawFrog()` for a hop rather than a slide.
- **submerged**: invisible (`drawFrog()` early-returns while in this state) for `SUBMERGED_MS` (2500ms) — genuinely underwater, not just off-screen.
- **returning**: same arc-hop interpolation back to the resting spot, firing `playSplash()` again, then back to `sitting` with the cooldown set.

`drawFrog()` is simple pixel art (body ellipse, two eye-bump circles, dark pupil dots) drawn at `(x, y - hop)` so the jump/return arc reads as a vertical hop layered on top of the horizontal slide.

`level-04.json`'s tree posts are placed as six blocks framing the yard (all 4 corners, plus one more each on the top and bottom edges), each pair at least 2 cells from its neighbors — trees only ever get trunk-radius collision (not a full `obstacleGrid` block, see Obstacle System above), so the mower can walk right up to and mow every cell around them, but a gap narrower than that could still pinch the mower between two trunks' collision circles. The pond sits in the deliberate gap left on the right side of the ring (no post there) rather than the ring being a perfectly even grid — fitting both a full ring and the pond within that level's 8-row height left no room for posts on *every* side without their 2-cell buffers colliding with the pond's, so the ring has an intentional opening rather than surrounding the pond on all sides. The pond itself is sized as large as that constraint allows with a 1-row grass margin top/bottom (30 cells, `levels/level-04.json`'s 6×5 `P` block) — an exhaustive search over every position/size in `buildObstacleLayer()`'s coordinate space (see git history around 2026-07-19) confirmed this is the maximum that still keeps a ≥2-cell gap from all six posts.

The posts themselves were later grown as large as they could go (36 tree cells total, up from the original 6×2×2=24) via the same kind of exhaustive per-post search, jointly and iteratively re-checking every other post's *current* (possibly already-grown) footprint each round rather than each post's original position — growing them independently first and then combining the results re-introduced new sub-2-cell gaps between posts that had each grown toward each other. Growth is always by 2 cells at a time in any direction (never 1), since tree sprites stamp one 32×32 texture per 2×2 sub-block within a cluster (`buildObstacleLayer()`) — an odd-sized cluster dimension would leave a sub-block stamped only half within the cluster's actual bounds, spilling visually into whatever's next to it.

`buildLevelTextures()` generates 4 `pond_corner_*` variants (`tl`/`tr`/`bl`/`br`) alongside the plain `pond` tile — the same water pattern (factored into a shared `drawPondBase()` closure) with a grass-colored (`C.bg`) triangular notch cut into the outer corner. `buildObstacleLayer()` stamps these at the 4 corners of a pond's bounding box instead of the plain tile (checking `dr`/`dc` against `cH - 1`/`cw - 1` within the same per-cell stamping loop bushes already use), so the overall shape reads as rounded rather than a sharp rectangle — collision (`obstacleGrid`) stays the full rectangle regardless; this is purely visual. Each pond cell also gets a single mutually-exclusive roll for a `lilypad` (~18% chance) or a `cattail` (~12%) stamped on top, so the two never clutter the same tile — both purely decorative, no effect on collision or mowability. (Cattails were first tried on the grass ring bordering the pond rather than in the water itself, but read better sitting in the pond alongside the lily pads.)

Squirrels normally run in a dead-straight line across the whole yard (see below) — on a pond level, a path whose row/column overlapped the pond would look like it was running across the water. `launchSquirrel()`'s `pondSafeValue()` picks a position outside the pond's (buffered) span on whichever axis matters for a given edge pair, and returns `null` if the pond's span (with buffer) swallows the *entire* usable range on that axis — which `level-04.json`'s pond does vertically, being 6 rows tall in an 8-row yard. When that happens, `launchSquirrel()` excludes that whole edge pair from consideration (here, left/right) rather than occasionally falling back to a value that still crosses — squirrels on that level always enter from the top or bottom edge as a direct consequence, never left/right.

## Audio
Procedurally synthesized via the raw Web Audio API (`AudioContext`/`OscillatorNode`/`GainNode`/`BiquadFilterNode`) rather than Phaser's sound manager (which is asset-based) or vendored audio files — no assets to source or host. Module-level (`g_audioCtx`, `g_humGain`), created once per page load by `setupAudio()`, not per `scene.restart()`. `setupAudio()` is called from the intro modal's Start button click specifically because that's a genuine user gesture, satisfying the browser's autoplay policy — calling it any earlier would leave the context stuck `suspended`.

Five sounds, all simple and gentle to match the game's low-key "mindless peacefulness" tone rather than reading as game-y sound effects (loud enough to actually register, though — an initial pass at gain 0.05 was nearly inaudible on typical speakers):
- **Mower hum**: a single continuous low-pass-filtered triangle oscillator, `start()`ed once and never stopped — oscillators can't be restarted, so movement on/off is expressed by ramping `g_humGain`'s gain (via `setHumActive()`, called every frame from `update()` with whether the player moved that frame) rather than starting/stopping the node, which also avoids audible clicks at the transition.
- **Win chime**: a short 4-note major arpeggio (`playWinChime()`, called from `showWin()`) — one short-lived oscillator+gain pair per note with a quick attack and exponential decay.
- **Bird chirp**: a quick two-note "tweet-tweet" (`playBirdChirp()`, called once from `launchBird()`), each note a fast upward pitch sweep in the 2-3kHz range — the classic chirp shape — with a very short envelope so it reads as a brief accent alongside the bird's flight, not an alert.
- **Dog bark**: a two-note "arf-arf" (`playDogBark()`, called once from `updateDog()` when the dog startles — see the Dog section above) — see its own description above for the square+lowpass detail.
- **Frog splash**: a quick "plop" (`playSplash()`, called from `updateFrog()` on both the jump-in and the jump-out) — a sine oscillator swept 500Hz→120Hz with a fast attack and short decay.

`setHumActive(false)` is called explicitly when the win condition triggers (in `updateHUD()`), since `update()` stops running entirely once `this.won` is true and would otherwise leave the hum wherever it last was.
